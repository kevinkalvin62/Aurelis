Objetivo

Traducir cada Musician Journey en una especificación de experiencia que sirva como puente entre la visión del producto y su implementación.

Cada especificación describirá el problema, el contexto, el resultado esperado y los principios que deberán respetarse antes de pensar en la interfaz.

1. Introducción
# Introducción

Una funcionalidad no representa el valor de un producto.

El valor aparece cuando una persona logra resolver un problema de forma sencilla.

Por ello, Aurelis no documentará módulos aislados.

Documentará experiencias completas.

Cada especificación de este documento nace de un Musician Journey.

Su propósito consiste en describir qué debe vivir el músico.

No cómo debe implementarse técnicamente.
2. Filosofía
Cada especificación responde una pregunta.

¿Qué debe experimentar el músico para que este momento se sienta natural?

El objetivo nunca será construir la mayor cantidad de funciones.

Será eliminar la mayor cantidad posible de fricción.

Cuando una implementación ofrezca varias alternativas, siempre elegiremos aquella que reduzca la carga mental del músico.

4. Ejemplo Oficial

Y aquí me gustaría dejar un ejemplo completamente desarrollado.

Experiencia
Preparar un ensayo
Contexto

Mañana existe un ensayo.

La agrupación ya conoce el repertorio.

El director necesita asegurarse de que todos lleguen preparados.

Problema actual

Los músicos buscan archivos en diferentes lugares.

Existen versiones distintas.

Cambios de tonalidad.

Mensajes dispersos.

El ensayo comienza resolviendo dudas.

No haciendo música.

Centralizar toda la información necesaria para que cada integrante llegue preparado.

5. Qué Nunca Documentaremos
No describiremos botones.

No describiremos colores.

No describiremos posiciones.

No describiremos componentes.

Esas decisiones pertenecen al Design System.

Este documento únicamente describe la experiencia que el producto deberá ofrecer.

7. Evolución
Las Experience Specifications evolucionarán junto con los Musician Journeys.

Toda nueva experiencia deberá justificar qué problema humano resuelve.

Las implementaciones podrán cambiar.

La experiencia esperada deberá permanecer.
8. Declaración de Permanencia
Cada experiencia documentada representa una promesa hecha al músico.

Podrán cambiar las tecnologías.

Podrán cambiar las pantallas.

Podrán cambiar los dispositivos.

Pero la experiencia que prometemos deberá mantenerse.

Porque los músicos no recuerdan componentes.

Recuerdan cómo un producto les ayudó a hacer mejor aquello que aman.
